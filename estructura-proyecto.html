Estructura del Proyecto Relatos de Papel

¿Qué son los Microservicios?

Los microservicios son un enfoque arquitectónico y organizativo para el desarrollo de software donde el software está compuesto por servicios pequeños e independientes que se comunican entre sí mediante APIs bien definidas. Cada servicio:

  • Se ejecuta en su propio proceso
  • Se comunica con otros servicios mediante mecanismos ligeros
  • Se despliega de forma independiente
  • Se organiza en torno a capacidades de negocio

Patrones de Diseño en Microservicios

Service Discovery (Eureka Server)

Permite que los servicios se encuentren y se comuniquen entre sí de forma dinámica. En nuestro caso, Eureka Server mantiene un registro de todos los servicios disponibles y sus ubicaciones.

API Gateway

Actúa como un punto de entrada único para todos los clientes, proporcionando enrutamiento, composición y transformación de peticiones. Implementa aspectos transversales como autenticación, SSL y rate limiting.

Circuit Breaker

Previene fallos en cascada cuando un servicio está fallando, permitiendo que el sistema siga funcionando de manera degradada en lugar de fallar completamente.

Este proyecto está estructurado como una aplicación de microservicios, donde cada componente tiene una responsabilidad específica. A continuación, se detalla la estructura y funcionalidad de cada componente:

1. Eureka Server (eureka-server)

Este es el servidor de registro y descubrimiento de servicios que:

  • Mantiene un registro de todos los microservicios disponibles
  • Permite la comunicación entre servicios
  • Facilita el balanceo de carga
  • Proporciona alta disponibilidad del sistema

2. API Gateway (api-gateway)

Actúa como punto de entrada único para todas las peticiones y:

  • Enruta las peticiones a los microservicios correspondientes
  • Implementa seguridad y autenticación
  • Centraliza la gestión de CORS
  • Proporciona un punto único de acceso para los clientes

3. Microservicio de Catálogo de Libros (ms-books-catalogue)

Gestiona toda la información relacionada con los libros:

  • Almacena información de libros (título, autor, ISBN, etc.)
  • Gestiona el inventario de libros
  • Proporciona búsqueda y filtrado de libros
  • Maneja las categorías y géneros literarios

4. Microservicio de Pagos (ms-books-payments)

Se encarga de toda la gestión de pagos y transacciones:

  • Procesa pagos de libros
  • Gestiona transacciones
  • Maneja diferentes métodos de pago
  • Proporciona historial de transacciones

Comunicación entre Microservicios

En nuestra arquitectura, los servicios se comunican de varias formas:

  • Comunicación Síncrona: Mediante REST APIs para operaciones que requieren respuesta inmediata
  • Comunicación Asíncrona: Mediante mensajes para operaciones que pueden procesarse en segundo plano
  • Eventos: Para mantener la consistencia eventual entre servicios

Beneficios de esta Arquitectura

  • Alta escalabilidad: Cada servicio puede escalar independientemente
  • Mantenibilidad: Los servicios son independientes y pueden ser actualizados sin afectar a otros
  • Resiliencia: Un fallo en un servicio no afecta al sistema completo
  • Flexibilidad: Permite usar diferentes tecnologías para cada servicio

Consideraciones de Diseño

Base de Datos por Servicio

Cada microservicio tiene su propia base de datos, lo que permite:

  • Independencia en el esquema de datos
  • Escalado independiente
  • Mejor aislamiento y seguridad

Despliegue Independiente

Los servicios pueden ser desplegados de forma independiente, lo que permite:

  • Actualizaciones más frecuentes
  • Menor riesgo en los despliegues
  • Mejor control de versiones
